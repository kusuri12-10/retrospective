# 멀티 프로세스 브라우저 아키텍처

## 프로세스와 스레드

- ***프로세스***는 운영체제(OS)로부터 메모리 영역을 할당받아 실행 중인 프로그램
- 프로세스 내부에는 실행 흐름인 ***스레드***가 존재한다.

- 2개 이상의 프로세스가 동시에 실행되는 구조를 ***멀티 프로세스***라고 부른다.

자바스크립트는 싱글 스레드 기반이지만, 비동기 처리를 통해 사용자에게는 여러 작업이 동시에 일어나는 것처럼 느껴지게 한다.

## 하드웨어(CPU 코어)와의 연관성
- 소프트웨어 영역의 스레드 하나는 하드웨어인 CPU의 코어 하나에 매핑되어 연산이 처리된다.

- 싱글 코어 프로세서는 한 번에 하나의 스레드만 처리할 수 있는 하위 구조를 가진다.

- 멀티 코어 프로세서는 여러 개의 코어를 가지고 있어, 여러 스레드를 동시에 물리적으로 처리할 수 있다.

## 프로세스 간 통신(IPC)과 메모리 구조
- 각 프로세스는 독립된 메모리 영역을 할당받기 때문에, 서로 다른 프로세스의 메모리 데이터에 직접 접근하는 것은 불가능

- **IPC(Inter-Process Communication)**: 프로세스끼리 데이터를 공유하기 위해서는 사용하는 별도의 통신 메커니즘

- 멀티 스레드들은 메모리 영역을 공유하므로 IPC를 거치는 것보다 훨씬 빠르게 데이터를 주고받을 수 있음

## 멀티 프로세스 아키텍처의 장단점 및 크롬 적용

- 장점: 
    - 각 프로세스가 격리된 메모리를 가지므로 접근 권한을 제한할 수 있어 보안성이 높음
    - 특정 프로세스 오류가 전체에 퍼지지 않아 안정적
- 단점: 
    - 프로세스를 여러 개 띄울수록 각각 독립적인 메모리 공간을 점유하게 되어 전체적인 메모리 사용량이 증가하게 됩니다

- 크롬 브라우저는 안정성과 보안을 위해 멀티 프로세스 아키텍처를 채택

## 크롬 브라우저 아키텍처의 구성 요소와 역할
    브라우저 프로세스, 렌더러 프로세스, GPU 및 플러그인 프로세스
- 브라우저 프로세스: 주소창, 뒤로 가기 버튼, 탭 닫기 등 브라우저의 UI를 담당하는 UI 스레드와 네트워크 요청을 처리하는 네트워크 스레드를 포함합니다
- 렌더러 프로세스: 웹사이트가 표시되는 탭 내부의 모든 화면 처리를 담당하며, 기본적으로 탭 하나당 하나의 프로세스가 생성됩니다
- GPU 및 플러그인 프로세스: 화면 전체를 그리는 역할을 하는 GPU 프로세스와 각 플러그인마다 개별적으로 생성되는 플러그인 프로세스가 존재합니다

## 멀티 프로세스 아키텍처의 주요 장점
- 시스템 안정성: 특정 탭(렌더러 프로세스)에서 응답 없음 오류가 발생하여 강제 종료하더라도, 다른 프로세스들은 독립적이기 때문에 나머지 탭들은 정상적으로 유지됩니다
- 보안 및 샌드박싱: 운영체제가 프로세스마다 고유한 메모리 영역을 할당하므로, 한 프로세스가 다른 프로세스의 데이터에 허락 없이 접근할 수 없어 보안에 유리합니다
- 사이트 격리(Site Isolation): 보안을 더욱 강화하기 위해 각 사이트별로 프로세스를 분리하며, 심지어 아이프레임(iframe) 하나당 별도의 프로세스를 생성하기도 합니다

한 프로세스는 다른 프로세스로 접근할 수 없음 -> 보안 강화 및 오류 발생 시 전파 안됨

## 아키텍처의 한계와 리소스 최적화 전략

- 메모리 사용량 증가: 프로세스가 늘어날수록 각각 독립적인 메모리 영역을 점유하게 되어 전체적인 시스템 메모리 소모가 커지는 단점이 있습니다
- 프로세스 수 제한: 크롬은 실행 중인 기기의 CPU와 메모리 상태를 계산하여 생성할 수 있는 프로세스의 개수에 한계치(Limit)를 둡니다
- 프로세스 통합(Consolidation): 기기의 리소스가 부족해지면 여러 개의 프로세스를 하나로 합쳐 메모리 영역을 공유하게 함으로써 메모리를 절약(Saving)하는 유연한 구조를 가집니다

멀티 프로세스: 메모리 사용량이 큼 -> 프로세스 수를 제한하거나 프로세스를 합치는 방식으로 메모리 관리
