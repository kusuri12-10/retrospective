## Reco 기획

### 프로젝트 기획서 작성

- HLS(HTTP Live Streaming) 기반의 영상 업로드 및 시청 플랫폼
- 핵심 목표: 영상 업로드 → 트랜스코딩 → HLS 스트리밍 파이프라인 구축
- mvp 목표: 2주 내 영상 업로드 및 HLS 재생 구동 기능 구현

### 기술 스택 선택

1. API 서버
    - Spring MVC: 익숙한 프레임워크를 선택하여 개발 속도 향상

2. worker 서버
    - Python + FFmpeg: 영상 처리, 다중 해상도 트랜스코딩
    - Celery + Redis: 비동기 작업 큐. 트래픽이 많아질 경우 Apache Kafka 변경 고려
    - HLS(HTTP Live Streaming): 네트워크 상태에 따라 화질 자동 조절

3. 프론트엔드
    - Next.js: SSR로 초기 로딩 빠름, SEO 유리
    - Video.js: 비디오 플레이어, 추후 HLS.js로 변경 고려
    - UI 컴포넌트: shadcn/ui + Tailwind CSS — 빠른 개발
    - useState + React Query: 상태 관리

4. 데이터베이스
    - PostgreSQL: 안정성, Full-text 검색, JSON 지원
        - 검색 기능은 필요할 경우 추후 Elasticsearch 도입
    - Redis: 
        - 캐시: 조회수, 세션, 인기 영상 캐싱
        - Pub/Sub: 실시간 조회수, 알림 -> 필요할 경우 kafka 도입 고려

5. 인프라
    - Kubernetes: 트랜스코딩 워커, API 서버 독립 스케일링
    - Prometheus + Grafana: 스트리밍 품질, 버퍼링률 모니터링
    - Cloudflare R2: 스토리지, 영상 송출에 소모되는 이그레스 비용 무료
    - Cloudflare: CDN, R2와 같은 네트워크 내부이기에 소모 비용이 낮음

### 요구사항 분석

1. 사용자 역할 (일반 사용자, 크리에이터, 관리자)

2. 기능 요구사항
    - 인증
    - 영상 업로드 및 처리
    - 영상 재생
    - 검색
    - 댓글 / 알림
    - 조회수 / 시청 기록
    - 관리자

3. 비기능 요구사항
    - API 응답시간: p95 기준 200ms 이하
    - 영상 초기 재생 시작: 3초 이내
    - 트랜스코딩 처리시간: 10분 영상 기준 5분 이내
    - 동시 스트리밍: 초기 목표 100명 동시 접속
    - 보안: Signed URL, JWT, bcrypt, Rate Limiting
    - 가용성: 트랜스코딩 실패 시 자동 재시도

### WBS (Work Breakdown Structure)

**S1**
- 인증
- 영상 업로드 파이브라인
- 영상 재생

**s2**
- 다중 해상도 지원
- CDN 최적화
- 검색

**s3**
- 조회수 / 시청 기록
- 댓글 / 알림
- 관리자 대시보드

**s4**
- 성능 최적화
- 인프라 구축
- 테스트 및 문서화

### 서비스 흐름도 작성

1. 영상 업로드 흐름
```
1. 클라이언트 업로드 요청
2. Presigned Url 발급
3. R2에 영상 업로드
4. 업로드 완료 및 신호 전송
5. 영상 메타데이터 저장 및 Celery 큐에 트랜스코딩 작업 등록
6. 영상 트랜스코딩 및 썸네일 생성
7. HLS 파일 및 썸네일 R2에 업로드
8. 영상 상태 업데이트
```

2. 영상 재생 흐름
```
1. 영상 상세 페이지 요청
2. 영상 정보 조회 및 Signed URL 생성
3. 반환값 SSR HTML 렌더링 후 브라우저에 전달
4. Video.js 플레이어로 HLS 재생
5. 캐시 HIT → 즉시 HLS 세그먼트 서빙 / 캐시 MISS → R2에서 가져와 캐싱 후 서빙
```
